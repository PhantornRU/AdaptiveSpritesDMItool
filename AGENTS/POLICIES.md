# Policies

## Стиль ответа
- Коротко и по делу.
- Сначала факты и риски, затем реализация и проверки.
- Для крупных изменений: план, реализация, проверка командами, остаточные риски.
- Не гадать там, где факт можно быстро проверить по репозиторию.

## Обязательный аудит перед изменениями
1. Определить тип scope: docs, UI/XAML, ViewModel/controller, processing, settings/path handling, build.
2. Проверить текущее состояние: связанные файлы, bindings, команды, глобальные флаги, side effects, ручные сценарии проверки.
3. Явно зафиксировать, что не меняется.
4. Для нетривиальной задачи подготовить decision-complete план без подхода "разберемся по ходу".
5. Сформулировать критерии приемки и команды проверки.

## Challenge-block для нетривиальных задач
Нетривиальные задачи: рефакторинг, перенос логики между слоями, изменение контрактов сохранения/экспорта, новые экраны, правки глобального состояния, архитектурные изменения.

Для таких задач обязательно включать:
1. Минимум 3 содержательных сомнения.
2. Проверку допущений: что подтверждено кодом, а что остается гипотезой.
3. Минимум 2 альтернативы реализации.
4. Обоснование выбранной альтернативы и причины отказа от остальных.
5. Список рисков и способ закрытия каждого риска.

No-Go:
- Если сомнения не закрыты фактами из репозитория или тестируемыми гипотезами, реализация не начинается.

## Принципы реализации
- Сохранять текущую границу между UI, состоянием приложения и обработкой файлов.
- Не раздувать глобальное состояние без необходимости.
- Предпочитать точечные изменения в существующих контроллерах и моделях вместо дублирующей логики.
- Для крупных рефакторингов не делать монолитную перепись за один заход; сначала декомпозировать работу на отдельные слои и понятные этапы.
- При изменении XAML не забывать проверять связанный ViewModel/API bindings и обратные вызовы из code-behind.
- При изменении путей, сериализации или пакетной обработки не ломать существующий формат пользовательских файлов и структуру экспорта.
- Не редактировать вручную generated/build artifacts, если это не прямой scope задачи.

## Анти-паттерны
- Править только XAML или только ViewModel, не проверив вторую сторону binding-контракта.
- Менять `EnvironmentController` или `StatesController` без аудита всех callsites.
- Пытаться провести большой архитектурный рефакторинг как один монолитный diff без промежуточного плана и без декомпозиции по слоям.
- Случайно менять семантику экспорта, override-логики или дефолтных директорий ради локального фикса.
- Опираться на temp/recovery файлы как на основной источник правды.
- Завершать задачу без build-проверки или без явного объяснения, почему она не запускалась.

## Definition of Done
Задача завершена, если:
1. Все решения внутри scope закрыты.
2. Изменения локальны, понятны и не размывают границы слоев.
3. Документация обновлена там, где поменялись правила или контракты.
4. Выполнены релевантные проверки или явно указано, почему они не запускались.
5. Итоговый ответ содержит список изменений, результаты проверок и остаточные риски, если они есть.
