# Рендеринг и оптимизация WPF UI

В Adaptive Sprites DMI Tool v2.2 используется переработанный пайплайн рендеринга редактора (рабочей области с сеткой) для устранения задержек при рисовании и зумировании.

## Переход на WriteableBitmap

В предыдущих версиях или наивных реализациях отрисовка пиксельной сетки могла выполняться через `DrawingContext.DrawRectangle` для каждого отдельного пикселя или ячейки. При высоких разрешениях (например, 128x128) или большом масштабе это создавало огромную нагрузку на WPF-рендер, так как формировалось огромное количество визуальных примитивов (visuals).

**Решение:** Был осуществлен переход на `WriteableBitmap`.
- Состояние поверхности (`EditorSurfaceRenderState`) хранит сырые данные пикселей в массиве `byte[] RgbaBytes`.
- При обновлении (рисование, заливка, стирание) изменяется только этот массив байтов в памяти.
- Затем массив целиком копируется в `WriteableBitmap.WritePixels`, обновляя изображение за одну операцию.
- Это радикально снижает нагрузку на CPU и память, позволяя рисовать практически без задержек.

## Управление состоянием (EditorSurfaceRenderState)

Класс `EditorSurfaceRenderState` инкапсулирует данные для рендеринга:
- Использует `byte[] RgbaBytes` для хранения цветовых значений пикселей.
- Это обеспечивает высокую скорость доступа и модификации, минуя накладные расходы WPF.
- После изменения пикселей состояние сигнализирует о необходимости перерисовки (через invalidate или аналогичный механизм), и измененный буфер отправляется в UI.

## Механизм LOD (Level of Detail) для текста

Помимо самих пикселей, на холсте отображаются подписи координат (Captions) для ячеек сетки. При сильном отдалении текста становится слишком много, он сливается и вызывает значительное падение производительности из-за рендеринга сотен мелких объектов `FormattedText`.

**Решения:**
1. **LOD (Level of Detail):** Текст координат скрывается (не рендерится), когда размер ячейки на экране становится меньше 12 логических пикселей (`cellSize < 12d`). Это мгновенно решает проблему производительности при сильном отдалении (zoom out).
2. **Кэширование `FormattedText`:** Объекты `FormattedText` не создаются заново на каждом кадре отрисовки `OnRender`. Они кэшируются (например, в словаре или переиспользуются), что снижает давление на сборщик мусора (GC) и уменьшает задержки при панорамировании.

## Батчинг обновления превью

Интерактивные инструменты (Paint, Erase) генерируют события изменений конфигурации непрерывно при движении мыши. Чтобы интерфейс не "зависал" от постоянных попыток пересобрать превью, применяется батчинг на уровне `MainWindowViewModel` или координатора:
- Обновления превью откладываются.
- Используется таймер с интервалом около 16 миллисекунд (что соответствует 60 FPS).
- Если за это время произошло несколько изменений, превью пересобирается только один раз.
- Детали описаны в [ARCHITECTURE.md](ARCHITECTURE.md).

## Imported DMI Layers

В v2.2 preview/editor composition также учитывает импортированные DMI state-слои:

- background layers смешиваются перед основным editable/base изображением;
- overlay layers смешиваются поверх результата;
- порядок слоев задается числовым `Order`;
- opacity каждого слоя нормализуется в диапазоне `0..100` и применяется при blending;
- кадры импортированных states кэшируются и инвалидируются при изменении source/assignment/order/opacity.
